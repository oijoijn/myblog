1. Dockerとは

Dockerを一言でまとめると,仮想環境のコンテナ化とその配布・管理を行うためのプラットフォームである.また,osや環境変数の違いを吸収してくれるため複数人での開発においてos,環境特有のエラーを防ぐことができる.
Dockerでは,仮想環境のことをコンテナと呼び,Docker Imageを元に作成される.

Docker用語

  • 仮想環境=コンテナ
  • Docker Image=コンテナの元になるもの,設定ファイル,スクリプト,などが含まれている

Dockerサービスの一つであるDocker Hubには様々なDocker Image(イメージ)が置かれており,Docker pullコマンドでイメージのダウンロードを行うかDockerfileを使用してオリジナルのイメージを作成することができる.

Dockerfileが必要になるタイミング
 例えば,Docker HubにあるDjnagoイメージを見に行くとpython 3.4が使用されておりとても古い.Dockerfileを使用することで最新のpythonや指定のパッケージで開発することができる.

Dockerfileはイメージのビルド手順を記述したファイルなので,Docker buildコマンドを使用することでDockerfileからイメージの作成することができる.
※大文字はDockerfileのコマンド

Dockerfileの例


      # syntax=docker/dockerfile:1
      FROM python:3.8                                          #元となるイメージ イメージ名:バージョンと書く,イメージ名だけだと最新versionが使用される
      WORKDIR /app                                             #コンテナ内の絶対パスからの作業ディレクトリを指定,mkdir -p のような動作
      COPY requirements.txt .                                  #指定のディレクトリやファイルをWORKDIRにコピー ホストosのpath コンテナ内のpath と書く
      RUN pip install --no-cache-dir -r requirements.txt       #コマンドを実行

整理
Docker pull = Docker Hubにあるイメージをlocalマシンにinstall
Dockerfile = オリジナルイメージを定義
Docker build = Dockerfileからイメージの作成を行う

概念図2

コンテナは原則として他のコンテナやホストOSにアクセスすることができない.つまり,コンテナ内にサーバーを立ててもホストos側からアクセスすることができない.また,コンテナを削除するとコンテナ内で行った変更がすべて取り消されるため,作成したファイルが失われる.それを防ぐため、Docker networkやマウントが使用される.

2. Mount

Dokcerのマウントには3種類あるがここでは2つ紹介する.バインドマウント(bind mount)とボリュームマウント(volume mount)である.これらは，主にコンテナ内の変更を維持するために存在するが使用用途の違いがある.

ボリュームマウントはデータベースに対して使用される.コンテナ削除後もデータを永続化することができコンテナ間でのデータ共有も可能である.マウントは,コンテナのディレクトリとDockerが管理しているストレージである.また,Docker Volumeはマウントしているコンテナを削除しても消えることがない.

バインドマウントはホストosとコンテナに対し使用される. ファイルの実行はコンテナ内で行うがファイルの編集をホストosで行いたいときにバインドマウントが使用される(コンテナ内にvscodeをinstallしていないため).

注意
マウントコマンドはtype=hoge, src というように空白を作るとエラーが起こる.
srcでホストosのpathまたはボリューム,targetでコンテナ上のpathを指定する.

Docker用語 バインドマウント = ホストosとコンテナをマウントする
ボリュームマウント = コンテナとdockerをマウントする

概念図3

何度も言うことになりますが,コンテナは原則として他のコンテナやホストOSにアクセスすることができない.ここではコンテナ間の通信を行えるようにする.

3. Docker Network

Docker network
Docker networkはコンテナ間の通信を可能にするもの.例えば,フロントエンドのコンテナ,バックエンドのコンテナ,データベースのコンテナを立てた際このままではお互いに通信できない.そこでDocker networkを使用することにより3つのコンテナ間の通信が可能になる.

4. Docker Compose

Docker Compose
Docker composeはdocker-compose.ymlファイルを使用する.Dockerfileがコンテナの定義を書くのに対しdocker-compose.ymlは設定やサービス間の連携を書く.また,Docker networkやDocker volumeを事前に作成していなくとも自動的に作成される.

docker-compse.ymlの例


      services:
        mysql:                                            # コンテナ名 自由に設定可能
          image: mysql:8.0                                # 使用するイメージ
          volumes:                                        # ボリュームマウント（データの永続化）
            - create_myblog-db:/var/lib/mysql             # ローカルのデータをMySQLのデータディレクトリにマウント
          ports:                                          # 使用するポート番号
            - 3306:3306
          environment:                                    # 環境変数（MySQLの設定）
            MYSQL_ROOT_PASSWORD: hogehoge
            MYSQL_DATABASE: hogehoge
            MYSQL_USER: hogehoge

        django:
          build: .                                        # オリジナルイメージを使用するためDockerfileのパスを指定
          command: /bin/bash
          volumes:                                        # バインドマウント（ホストとコンテナでディレクトリを共有）
            - path:path
          ports:                                          # Djangoのアプリケーションに対するポートマッピング
            - 127.0.0.1:8000:8000
          tty: true
          depends_on:                                     # mysqlコンテナが先に起動するように依存関係を指定
            - mysql

      volumes:
        create_myblog-db:                                 # ボリュームの定義（MySQL用データの永続化に使用）

Docker networkコマンド 


      docker compose up -d        #複数コンテナを立てる場合バックグラウンドで実行しdocker execコマンドで個別にアクセス
      docker compose down         #コンテナの削除 ※ボリュームは削除されないためsqlデータは保持される

注意
docker-compse.ymlに直接環境変数を書くのはセキュリティ上よくないのでenvファイルから読み込むようにする.

5. まとめ

Docker Hubからイメージをダウンロードし,必要であればDockerfileを使って独自のイメージを作成しコンテナを起動する.コンテナ同士で通信するためにDocker Networkを使用し、コンテナ内で利用したデータベースなどのデータを永続化するためにDocker Volumeを使用する.
